## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

自動実装の Verification フェーズにおいて、Verification エージェントが E2E テスト仕様で指定された CLI コマンドを実行せず、ユニットテスト結果を代用して全 PASS と報告する事象が発生した。現行の品質ゲート構成では RFC レビューと実装レビューにゲートキーパーが存在するが、Verification 結果にはゲートレビューが不在であり、エージェントの自己申告がそのまま最終結果となる構造的欠陥がある。承認済み仕様の不変性保証は対応済みであるが、検証結果の捏造防止はそれとは直交する未解決の問題である。

| ID | 要件 |

## 6. 外部仕様 (External Specification)

<!-- ユーザ・運用者視点の振る舞い。箇条書き＋コード例。 -->

- Verification ゲートレビューコマンド実行時、
  検証結果ファイルに対してキーワードスキャンと
  ゲートキーパー検証を実行する
- キーワードスキャンで禁止パターン
  （ユニットテストファイル参照、
  テストフレームワーク用語等）を検出した場合、
  該当ゲートを FAIL に上書きする
- ゲートキーパーは仕様の実行手順と
  エビデンスの突合を意味的に検証する
- 結果は既存レビューと同じ形式で
  レビュー結果ファイルに出力する
- GATE-V1 FAIL 時の出力形式:

```
## GATE-V1: 結果値の健全性
**判定: FAIL**
- NG: （不正な結果値の引用）
- 対応: （FAIL 時の固定対応文）
```

- GATE-V2 FAIL 時の出力形式:

```
## GATE-V2: 実行証跡の真正性
**判定: FAIL**
- NG: （不正なエビデンスの引用）
- 対応: （FAIL 時の固定対応文）
```

- 自動実装コマンドの Verification ループでは、
  Verification 実行後に自動でゲートレビューが実行される
- ゲートレビュー FAIL 時は修正ループに入り、
  最大8回まで再試行する
- 自動開発スクリプトの全体 E2E テストでも同様に
  ゲートレビューが自動実行される

### 変更1: Verification ゲート検証項目の新設

`prompts/criterias/gate-criteria-vfy.md` を以下の内容で新設する。

```markdown
# Verification ゲートキーパー検証項目

## GATE-V1: 結果値の健全性

### 基準 A（適合証明）
- verification-results.md の「結果」列が
  PASS または FAIL のみで構成されているか

### 基準 B（不適合排除）
- SKIP、N/A、その他の不正な値が
  1つでも存在しないか

### FAIL 時の固定対応文

不正な結果値を検出。
検証結果の「結果」列は PASS / FAIL の
2値のみが許可される。
SKIP・N/A・その他の値を使用した
テスト項目を再実行し、
PASS または FAIL で報告せよ。

## GATE-V2: 実行証跡の真正性

### 基準 A（適合証明）
- エビデンス列の記述が、仕様の実行手順で
  指定されたコマンドの実際の実行結果であるか
- RFC §7（または仕様書 §2）の実行手順と
  エビデンスの対応が取れているか

### 基準 B（不適合排除）
- ユニットテストファイル参照
  （.test.ts, .test.js, .spec.ts, .spec.js）が
  エビデンスに含まれていないか
- テストフレームワーク用語
  （vi.fn, vi.mock, jest.fn, jest.mock, vitest）が
  エビデンスに含まれていないか
- モック関連用語
  （モック, スタブ, スパイ）が
  エビデンスに含まれていないか
- 関数レベル直接呼び出し
  （直接実行:）が
  エビデンスに含まれていないか
- SKIP / N/A が
  エビデンスに含まれていないか

### FAIL 時の固定対応文

不正な実行証跡を検出。
エビデンスは仕様で指定された
実行手順の実際の実行結果でなければならない。
ユニットテスト結果の代用・
モックによる検証・
関数の直接呼び出しは許可しない。
仕様の実行手順に従い E2E テストを再実行し、
実際のコマンド出力をエビデンスとして記録せよ。
```

### 変更2: Verification ゲートレビューコマンドの新設

`adapters/claude/commands/rvfy.md` を以下の内容で新設する。

```markdown
# /rvfy - Verification ゲートレビューコマンド

指定された slug の Verification 結果に対し、
2層ゲートキーパー（キーワードスキャン +
ゲートキーパー判定）による検証を実行せよ。

## 対象

$ARGUMENTS

## 実行モード

本コマンドは2つのモードで動作する。
$ARGUMENTS の内容で自動判定する。

### RFC モード（既定）

$ARGUMENTS が slug の場合に適用する。
検証結果ファイル: docs/rfcs/<slug>/verification-results.md
仕様ファイル: docs/rfcs/<slug>/rfc.md（§7 E2Eテスト仕様）

### service-spec モード

$ARGUMENTS が仕様書パスの場合に適用する。
検証結果ファイル: 仕様書と同ディレクトリの verification-results.md
仕様ファイル: 指定された仕様書（§2 E2Eテスト要件）

## 実行手順

### Step 1: 対象の取得とモード判定

上記「対象」が空の場合は、
「検証対象のslugまたは仕様書パスを入力してください。」
とだけ表示し、ユーザの次のメッセージを待て。

$ARGUMENTS がファイルパスとして存在する場合は
service-spec モード、それ以外は RFC モードとする。

### Step 2: ブランチ確認

- RFC モード: 現在のブランチが feature/<slug>
  であることを確認せよ。
  異なる場合は feature/<slug> にチェックアウトせよ。
- service-spec モード: ブランチ確認は行わない。

### Step 3: 検証結果ファイル存在確認

カレントリポジトリのルート
（git rev-parse --show-toplevel）を基準に
検証結果ファイルの存在を確認せよ。
ファイルが存在しない場合はエラーを報告して終了せよ。

検証結果ファイルと仕様ファイルの絶対パスを
控えておくこと（Task に渡すため）。

注意: 検証結果ファイルと仕様ファイルを

### Step 4: ラウンド判定

- RFC モード:
  docs/rfcs/<slug>/ 配下の review-vfy-r*.md
  ファイル数を確認し、ラウンド番号を自動判定する。
- service-spec モード:
  仕様書と同ディレクトリの review-vfy-r*.md
  ファイル数を確認し、ラウンド番号を自動判定する。

- review-vfy-r*.md が 0件 → ラウンド1
- review-vfy-r*.md が N件 → ラウンド N+1

「レビューラウンドが上限（8回）に達しました。
手動で対応してください。」と報告して終了せよ。

禁止用語リスト:

.test.ts, .test.js, .spec.ts, .spec.js,

モック, スタブ, スパイ,

### Step 5: ゲートキーパー検証実行

以下のプロンプトで Task を起動し、

2. 検証項目: {gate-criteria-vfy.md の絶対パス}
3. 検証対象 Verification 結果: {検証結果ファイルの絶対パス}
4. 仕様ファイル: {仕様ファイルの絶対パス}

## 指示
- 人格定義の判定原則に従い、検証項目に定義された
  すべてのゲートを順に検証せよ。
- GATE-V2 では、仕様ファイルの E2E テスト仕様

  両方を判定せよ。

### Step 5.5: キーワード検出による FAIL 上書き

Step 4.5 でキーワードが検出された場合、
以下のキーワード→ゲート対応表に基づき、
該当ゲートの判定を機械的に FAIL に上書きせよ。

上書き時は、該当ゲートの「対応」行のみを
検証項目に定義された「FAIL 時の固定対応文」
テキストで置換する。
NG 行はゲートキーパーの出力をそのまま維持する。

ゲートキーパーが当該ゲートを PASS 判定
している場合は、ゲート全体を以下の形式で
FAIL に上書きする:

## GATE-X: （ゲート名）
**判定: FAIL**
- NG: キーワードスキャンにより検出
- 対応: （検証項目の「FAIL 時の固定対応文」を転記）

ゲートキーパーが当該ゲートを既に FAIL 判定
している場合は、「対応」行のみを
固定対応文で上書きする。

#### キーワード→ゲート対応表

| キーワード群 | 対応ゲート |
|-------------|-----------|
| SKIP, N/A | GATE-V1 |
| .test.ts, .test.js, .spec.ts, .spec.js, | GATE-V2 |
| vi.fn, vi.mock, jest.fn, jest.mock, vitest, |  |
| モック, スタブ, スパイ, 直接実行: |  |
```
