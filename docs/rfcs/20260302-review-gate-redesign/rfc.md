## 1. 背景・動機 (Motivation)

gale リポジトリの E2E テストにおいて、モック HTML が実サイトの URL 形式を再現せず本番エラーが発生した。原因分析の結果、3人格レビューが些末な指摘に終始し本質的問題を検出できないこと、禁止用語スキャンが対象セクション限定で回避可能であること、要件の無断先送りや環境タスク欠落を防ぐ仕組みが存在しないことが判明した。一方、サービス仕様書コマンドで導入した Three-Tier Method（段階的導出手法）は、作成手順を正しく明示すれば作成者自身で十分な品質の成果物を作れることを実証した。本 RFC はこの知見に基づき、「作成時の品質組み込み」と「検証時の Default-Deny ゲート」による多層防御構造へ移行する。

## 2. 機能要件

### 達成すべき要件

| ID | 要件 |
|----|------|
| FR-01 | RFC 作成コマンドの E2E テスト仕様導出に段階的導出手法を組み込む |
| FR-02 | RFC レビューの3人格並列レビューを廃止し、2層ゲートキーパーに置換する |
| FR-03 | 2層ゲートキーパーの第1層はRFC全文に対する禁止用語スキャンとする |
| FR-04 | 2層ゲートキーパーの第2層は適合証明と不適合排除の2軸で判定する |
| FR-05 | ゲートキーパーは3つの独立ゲートで判定する（E2Eテスト品質・先送り排除・環境タスク欠落） |
| FR-06 | 実装レビューコマンドと実装修正コマンドを廃止する |
| FR-07 | 検証コマンドに RFC モードと service-spec モードの2モード対応を追加する |
| FR-08 | 自動実装コマンドから実装レビューループを削除し、検証ループのみに簡略化する |
| FR-09 | ゲートキーパーに前回レビュー情報を一切渡さない（イテレーション情報の完全遮断） |
| FR-10 | 自動開発コマンドに全体E2Eテストフェーズを追加する（全RFC完了後に /vfy を service-spec モードで実行し、FAIL 時は fix ブランチ作成→修正ループ→PR マージの GitHub Flow に従う） |
| FR-11 | サービス仕様書コマンドの §4.3 起草手順から「全体E2Eテスト実施RFC」の指定を削除する |
| FR-12 | サービス仕様書テンプレートの §4.3 から全体E2Eテスト行を削除する |

### やらないこと

- 実装コマンド自体の変更（RFC に基づく実装フローは現行維持）
- RFC 修正コマンドの変更（修正フローは現行維持）
- PR コメント対応コマンドの変更
- 自動 RFC レビューコマンドの変更

## 4. 実現方式

| 要件 | 実現方式 |
|------|----------|
| FR-01 | RFC 作成コマンドの E2E テスト仕様セクション起草前に4段階導出ステップを追加する |
| FR-02 | RFC レビューコマンドの並列3タスク + シンセサイザーを、キーワードスキャン + 単一ゲートキーパータスクに置換する |
| FR-03 | 禁止用語リストを RFC 全文に対して適用し、検出結果を第2層への確定的証拠として渡す |
| FR-04 | ゲートキーパー人格定義に適合証明基準と不適合排除基準を明記し、両基準の AND 条件でのみ PASS とする |
| FR-05 | ゲートキーパー検証項目に3ゲートの基準 A・基準 B を独立定義する |
| FR-06 | 実装レビューコマンド・実装修正コマンドのファイルを削除する |
| FR-07 | 検証コマンドに2モード対応（slug → RFC §7、ファイルパス → service-spec §2）を追加し、モードに応じたブランチ確認・結果出力・コミット処理を実装する |
| FR-08 | 自動実装コマンドから実装レビューフェーズを削除し、検証ループ最大5回に変更する |
| FR-09 | ゲートキーパータスクのプロンプトにレビュー成果物パス・ラウンド番号・差分情報を含めない |
| FR-10 | 自動開発コマンドの Phase 3 完了後に Phase 4（全体E2Eテスト）を追加する。初回 PASS 時はブランチ不要、FAIL 時は fix/e2e-* ブランチ作成→修正ループ（最大5回）→PR マージの GitHub Flow に従う |
| FR-11 | サービス仕様書コマンドの Phase 4 から「最終行は全体E2Eテスト実施RFC」を削除し、Phase 5 チェックリストから対応項目を削除する |
| FR-12 | サービス仕様書テンプレートの §4.3 コメントと全体E2Eテスト行を削除し、RFC 一覧を純粋な実装 RFC のみとする |
| NFR-01 | 廃止対象ファイルを Task 計画に削除タスクとして明記する |
| NFR-02 | ワークフロー定義の Stage 2・Stage 3 の記述を改定後の構成に更新する |
| NFR-03 | キーワードスキャン検出を基準 B 違反確定としてハードコードし、LLM の例外解釈を禁止する指示を明記する |

## 5. 代替案の検討

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: 2層ゲートキーパー + 作成時品質組み込み | 3人格を廃止し、Default-Deny の1ゲートキーパーに集約。RFC 作成時に段階的導出手法で品質を組み込む | **採用** | 分析レポートで3人格が本質的問題を検出できないことが実証済み。作成時の品質組み込みと検証時のゲート強化の二正面で防御する |
| B: 3人格レビューの検証項目強化 | 既存3人格を維持し、検証項目にモック禁止・先送り禁止・環境タスク確認を追加する | 却下 | 3人格の責務分割が些末な指摘を生む構造的原因であり、検証項目を追加しても各人格の視野制限は解消されない。再レビュー時のゲート強度低下も未解決 |
| C: 人間レビューへの回帰 | AI レビューを廃止し、全レビューを人間が行う | 却下 | 本プロジェクトの目的が AI 駆動開発の確立であり、人間レビューへの回帰は目的と矛盾する |

## 6. 外部仕様 (External Specification)

- RFC 作成コマンド実行時、E2E テスト仕様の導出に4段階の中間ステップが追加される
  - 中間成果物（機能列挙・条件軸・テストデータの各テーブル）が RFC 付録として残る
  - 自己チェックリストによる品質確認が DoD に追加される
- RFC レビューコマンド実行時の出力が変更される
  - 3つのレビューファイル + アクションプランが、1つのゲートレビューファイルに集約される
  - 出力ファイル名が `review-gate-r{N}.md` に変更される
- 検証コマンドが RFC モードと service-spec モードの2モードに対応する
  - slug 指定時は RFC §7 に基づく個別検証（従来動作）
  - ファイルパス指定時は service-spec §2 に基づく全体E2E検証
- 自動開発コマンド実行時、全 RFC 完了後に全体E2Eテストが自動実行される
  - /vfy が service-spec モードで仕様書 §2 の全テスト項目を検証する
  - 初回 PASS 時はブランチ作成なしで完了する
  - FAIL 時は `fix/e2e-*` ブランチ上で修正ループ（最大5回）を実行し、PR マージまで自動化される
- サービス仕様書コマンドの §4.3 RFC 一覧から「全体E2Eテスト実施」RFC の指定が撤廃される
  - 全体E2Eテストは /adev が自動実行するため、RFC として起草不要
- 自動実装コマンド実行時のフローが簡略化される
  - 実装レビューループが廃止され、検証ループ（最大5回）のみとなる
- 実装レビューコマンドと実装修正コマンドが使用不可となる

## 11. 詳細設計 (Detailed Design)

### 11.1 ゲートキーパー人格定義

`prompts/roles/gatekeeper.md` に以下の内容を定義する。

```markdown
# Role: Gatekeeper

Default-Deny の姿勢で RFC を検証する人格である。
「この RFC は不合格である」を出発点とし、
合格の積極的証拠が揃った場合にのみ PASS を許可せよ。

## 判定原則

- 基準 A（適合証明）: 適合の積極的証拠があるか
  - 証拠なし/不十分 → FAIL
- 基準 B（不適合排除）: 不適合の兆候が一切ないか
  - 少しでも疑わしい兆候 → FAIL
- PASS 条件: 基準 A 充足 AND 基準 B 不在

## キーワードスキャン検出時のルール

キーワードスキャンで禁止用語が検出された場合、
それは基準 B 違反の確定的証拠である。
以下を厳守せよ:

- 検出 = 基準 B 違反確定 → 無条件 FAIL
- 「文脈上問題ない」「否定文で使用されている」
  等の例外解釈は一切禁止
- 検出事実に対する LLM の独自判断を禁止

## フィードバック出力規則

- FAIL 時は「適合証拠の不足」を主軸に指摘せよ
- 「禁止語 X が検出された」という指摘は禁止
  （修正者が言い換えで回避することを防止）
- 「PASS に必要な修正」として具体的な記述追加を示せ
```

### 11.2 ゲートキーパー検証項目

`prompts/criterias/gate-criteria-rfc.md` に以下の3ゲートを定義する。

```markdown
# RFC ゲートキーパー検証項目

## GATE-1: E2E テスト品質

### 基準 A（適合証明）
- 各テストケースに外部サービスへの実接続が明記されているか
- テストデータがドメイン固有の実在データであるか
- テストシナリオがユーザ操作起点→最終アウトカム確認の
  構造であるか
- 各機能要件に対し正常系・異常系/境界値の2ケース以上
  あるか

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- テストデータに汎用ダミーデータの兆候がないか
- 中間処理の確認で終わるテストケースがないか
- セットアップ手順に環境準備が欠落していないか

## GATE-2: 勝手な先送り排除

### 基準 A（適合証明）
- 機能要件の全項目が設計・Task 計画で
  対応されている証拠があるか
- Task 計画が単一 PR で完遂可能な構成であるか

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- 要件の無断除外・先送りの兆候がないか
- Task 計画の分割・段階化の兆候がないか

## GATE-3: 環境状態変更タスクの欠落

### 基準 A（適合証明）
- 設計上必要な環境状態変更（DDL、cron、デプロイ設定、
  システムサービス設定、環境変数設定等）が
  Task 計画に含まれているか
- 環境状態変更が不要な場合、その根拠が明確か

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- 設計に環境状態変更を要する要素があるにもかかわらず
  Task 計画に対応タスクが欠落していないか
```

### 11.3 RFC 作成コマンドへの段階的導出手法追加

### 11.5 検証コマンドの2モード対応

`adapters/claude/commands/vfy.md` に以下の2モードを追加する。

#### モード判定ロジック

`$ARGUMENTS` がファイルパスとして存在する場合は service-spec モード、それ以外は RFC モードとする。

#### RFC モード（既定）

- ブランチ確認: `feature/<slug>` であることを確認し、異なる場合はチェックアウト
- E2Eテスト仕様読み込み: `docs/rfcs/<slug>/rfc.md` の §7
- 結果ファイル: `docs/rfcs/<slug>/verification-results.md`
- PR body 更新: 実行する
- コミット・プッシュ: 実行する

#### service-spec モード

- ブランチ確認: 行わない（デフォルトブランチ上で実行）
- E2Eテスト仕様読み込み: 指定された仕様書ファイルの §2
- 結果ファイル: `docs/verification-results.md`
- PR body 更新: スキップ
- コミット・プッシュ: スキップ

### 11.9 自動開発コマンドへの全体E2Eテストフェーズ追加

- Step 4-5: 全 PASS → PR を Ready にしてスカッシュマージ、デフォルトブランチに復帰
- Step 4-6: 5回到達しても FAIL → fix ブランチ名をユーザに報告して停止
