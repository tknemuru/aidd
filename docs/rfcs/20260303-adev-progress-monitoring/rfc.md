## 1. 背景・動機 (Motivation)

自動開発コマンドは外部シェルスクリプトを介して複数の自律プロセスを直列実行するが、各プロセスの実行中は進捗が一切可視化されない。実行時間が10分から60分超に及ぶため、進行中なのか停滞しているのかを判別できない運用課題がある。また、リポジトリ固有の安全監視をコマンド実行時に追加する拡張ポイントが存在しない。2026年3月2日に発生した外部サイトへの過剰リクエストインシデントを契機に、コード側の防御策に加えて実行監視層での多層防御が必要と判断した。

## 2. 機能要件

### 達成すべき要件

| ID | 要件 |
|----|------|
| FR-1 | 自動開発コマンド定義を4フェーズ構造に書き換える |
|       | - 初期化: 仕様書読み込み、slug一覧抽出、状態判定、監視ルール読み込み |
|       | - バックグラウンド起動: シェルレベルで非同期実行しPIDとログパスを記録 |
|       | - モニタリングループ: 適応型ポーリングによるリアルタイム監視 |
|       | - 完了レポート: 最終結果の報告 |
| FR-2 | ポーリング間隔を状態変化に応じて動的に調整する |
|       | - 初回/状態変化直後: 60秒 |
|       | - 無変化1回連続: 120秒 |
|       | - 無変化2回連続: 180秒 |
|       | - 無変化3回以上連続: 300秒（上限） |
| FR-3 | 各ポーリングサイクルで診断データを収集し状態判定・報告を行う |
|       | - プロセス生存確認、ログ末尾、決定ログ末尾、git/GitHub状態 |
|       | - 7段階のフェーズ判定（完了/実装PR/実装中/RFC完了/RFC PR/RFC作成中/待機） |
| FR-4 | リポジトリ固有の監視ルール拡張ポイントを提供する |
|       | - 規約ベース自動検出: 所定パスのファイルが存在すれば読み込んで従う |
|       | - フォーマット・構造を一切規定しない自由形式 |
| FR-5 | brillat リポジトリ用のバースト検知監視ルールを作成する |
|       | - 外部リクエスト送信パターンの監視 |
|       | - 異常判定: 連続リクエスト間隔2秒未満、または30秒間に10件以上 |
|       | - ユーザ承認なしの即座のプロセスグループ強制停止 |

### やらないこと

- 自動開発オーケストレータ本体の変更（シェルスクリプトは変更しない）
- git ユーティリティライブラリの変更
- 監視ルールファイルのフォーマット標準化やスキーマ定義
- brillat 以外のリポジトリ用監視ルールの作成

## 5. 代替案の検討

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: コマンド定義内ポーリング | コマンド定義にポーリングループを直接記述し、実行エージェントが監視を担う | **採用** | エージェント自身がモニタとして動作するため追加インフラ不要。コマンド定義の自然言語で柔軟な判定ロジックを記述できる |
| B: 別プロセス監視デーモン | 専用の監視スクリプトをバックグラウンドで並行起動する | 却下 | プロセス管理の複雑化、ログの分散、エージェントとの状態共有の困難さ。シェルスクリプト変更禁止の制約にも抵触する可能性 |
| C: ログファイルのtail監視のみ | ポーリングせずログファイルのtailだけ表示する | 却下 | フェーズ遷移やPR状態の構造化された報告ができない。適応型間隔制御も不可能 |

## 6. 外部仕様 (External Specification)

- 自動開発コマンドの使用方法は変更なし
  - 従来通り仕様書パスとセクション名を引数に指定して実行する
- コマンド実行中の挙動が変化する
  - オーケストレータがバックグラウンドで非同期実行される
  - エージェントが定期的に進捗状況をユーザに報告する
  - 状態変化がない場合はポーリング間隔が段階的に延長される
  - 状態変化を検知すると間隔が短縮され詳細報告が行われる
- リポジトリ固有の監視が自動的に有効化される
  - 所定パスにファイルを配置するだけで追加監視が動作する
  - brillat では外部リクエストのバースト検知が有効になる
- 異常検知時はプロセスが即座に停止される
  - バースト検知の場合、ユーザ承認なしで強制停止が実行される
  - 検知データがユーザに報告される

## 11. 詳細設計 (Detailed Design)

### 自動開発コマンド定義の構造

コマンド定義は以下の4フェーズで構成する。

- Phase 1: 初期化
  - 仕様書パスと対象セクションをコマンド引数から取得する
  - 仕様書を読み込み、指定セクションから RFC slug 一覧を抽出する
  - 各 slug の現在のフェーズを git/GitHub 状態から判定する
  - リポジトリ固有監視ルールファイルの存在を確認し、あれば読み込む
- Phase 2: バックグラウンド起動
  - オーケストレータを `nohup` とバックグラウンド実行で起動する
  - PID をシェル変数に記録する
  - ログ出力先のファイルパスを記録する

```bash
nohup ~/projects/vdev/bin/adev.sh "<仕様書パス>" "<セクション名>" \
  > /tmp/adev-<timestamp>.log 2>&1 &
ADEV_PID=$!
LOG_FILE="/tmp/adev-<timestamp>.log"
```

- Phase 3: モニタリングループ
  - 以下の手順をオーケストレータ終了まで繰り返す

```bash
# プロセス生存確認
kill -0 $ADEV_PID 2>/dev/null

# ログ末尾確認
tail -20 "$LOG_FILE"

# 決定ログ末尾確認（パスは仕様書ディレクトリ内の adev-decisions.md）
tail -10 "<spec-dir>/adev-decisions.md"

# git/GitHub 状態確認
git log --oneline -5
git branch -a | grep -E '(rfc|feature)/'
gh pr list --state all --limit 20
```

適応型ポーリング間隔の制御ロジックは以下の通りである。

```
no_change_count = 0

loop:
  collect diagnostic data
  detect state changes (phase transition, new commit, new PR, new file)

  if state changed:
    no_change_count = 0
    report detailed changes
  else:
    no_change_count += 1
    report brief status

  interval = lookup(no_change_count)
    0     -> 60s
    1     -> 120s
    2     -> 180s
    >= 3  -> 300s

  sleep $interval
  if process exited: break
```

状態変化の定義は以下のいずれかに該当する場合である。

  - slug のフェーズが前回と異なる値に遷移した
  - 新しいコミットが出現した（`git log` のHEADハッシュが変化）
  - 新しい PR が作成またはマージされた
  - 新しいファイルが出現した

フェーズ判定テーブルは以下の通りである。

| フェーズ | 判定条件 |
|---------|---------|
| 完了 | `feature/<slug>` の PR が MERGED |
| 実装PR | `feature/<slug>` の PR が OPEN |
| 実装中 | `feature/<slug>` ブランチが存在し PR 未作成 |
| RFC完了 | `rfc/<slug>` の PR が MERGED かつ feature 未着手 |
| RFC PR | `rfc/<slug>` の PR が OPEN |
| RFC作成中 | `rfc/<slug>` ブランチが存在し PR 未作成 |
| 待機 | 上記いずれにも該当しない |

報告ルールは以下の4パターンである。

  - 状態変化あり: 具体値（コミットハッシュ、PR番号、ファイル数等）を含む詳細報告
  - フェーズ遷移あり: 全 slug の進捗テーブルを表示
  - 状態変化なし: 「変化なし、次回確認まで N 秒」の簡潔報告
  - 異常終了検知: ログ末尾を表示して監視を終了

- Phase 4: 完了レポート
  - オーケストレータの終了コードを確認する
  - 全 slug の最終フェーズ判定を行い、進捗テーブルを表示する
  - 成功/失敗をユーザに報告する

### 監視拡張ポイントの設計

コマンド定義の Phase 1 に以下の1文を記載する。

```
対象リポジトリの .claude/monitors/adev.md が存在する場合は Read で読み込み、
Phase 3 のポーリングループで追加の監視指示として従え。
```

この設計により以下を実現する。

  - vdev 側は存在チェックと読み込み指示のみを定義する
  - ファイルのフォーマットやセクション構造を一切規定しない
  - 各リポジトリが自由形式で監視ルールを記述する
  - エージェントがインタプリタとして自然言語の指示を解釈・実行する

### brillat 用バースト検知監視ルール

`~/projects/brillat/.claude/monitors/adev.md` に以下の監視内容を記述する。

  - 監視対象: オーケストレータのログ出力中の `外部リクエスト送信` パターン
  - 異常判定条件（OR）:
    - 連続するリクエストのタイムスタンプ間隔が2秒未満
    - 直近30秒間のリクエスト数が10件以上
  - 異常検知時のアクション:
    - ユーザ承認なしでオーケストレータのプロセスグループを即座に kill する
    - 検知したデータ（タイムスタンプ、リクエスト数、間隔）を報告する
  - 経緯: 2026-03-02 の過剰リクエストインシデントへの多層防御

### バックグラウンド実行の技術的詳細

`nohup &` を使用する理由は以下の通りである。

  - Claude Code の `run_in_background` には無限 system-reminder ループの既知バグがある
  - シェルレベルの `nohup &` でこの問題を回避する
  - `$!` で PID を取得し、`kill -0` で生存確認を行う

プロセスグループの強制停止は以下のコマンドで実行する。

```bash
kill -- -$(ps -o pgid= -p $ADEV_PID | tr -d ' ')
```
