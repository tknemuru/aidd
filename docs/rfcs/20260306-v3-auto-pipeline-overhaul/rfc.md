## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

vdev 2.0 の自動導線は起点が3コマンドに分散し、構造が複雑化している。自動RFC・自動実装・統合開発の3コマンドが存在し、統合開発が自動RFCを経由してRFC作成を呼ぶN層ネスト構造のため、デバッグや障害復旧が困難である。また冪等性が「マージ済みスキップ」レベルに留まり、レビューループ途中からの再開ができない。本RFCでは自動導線を統合開発コマンドに一本化し、フラットな呼出構造・フェーズ単位の冪等性・リスクレベルによるフロー分岐を導入する。

## 2. 機能要件

### 達成すべき要件

<!-- コード識別子禁止。機能名称またはドメイン用語で記述。 -->

| ID | 要件 |
|----|------|
| FR-1 | 自動RFCコマンドと自動実装コマンドを廃止する |
| FR-2 | 自動導線の起点を統合開発コマンドに一本化する |
| FR-3 | オーケストレータがRFC作成等の各コマンドを直接呼出す |
| FR-3a | N層ネスト（統合開発→自動RFC→RFC作成）を排除する |
| FR-4 | 統合開発コマンド定義側でフェーズ判定を行う |
| FR-4a | シェルによる機械的状態チェックで判定する |
| FR-4b | 判定結果をフェーズマップ JSON としてユーザに提示する |
| FR-4c | ユーザ承認後にフェーズマップをオーケストレータに渡す |
| FR-4d | オーケストレータはフェーズ判定を行わない |
| FR-4e | 全8状態（RFC～DONE）のフェーズを判定する |
| FR-4f | レビューループ途中の判定不能時は再レビューから再開する |
| FR-5 | サービス仕様書テンプレートにリスクレベル欄を追加する |
| FR-5a | 仕様書作成コマンドでリスクレベル引数を受付ける |
| FR-5b | 引数未指定時はユーザに確認してから起草を開始する |
| FR-6 | リスクレベル高でゲート FAIL 時に人間エスカレーションする |
| FR-6a | リスクレベル高でRFC/実装PRマージ前に人間承認を得る |
| FR-6b | リスクレベル高で検証完了時に人間承認を得る |
| FR-7 | リスクレベル中でゲート FAIL 時に人間エスカレーションする |
| FR-7a | リスクレベル中でゲート通過時は自動マージを許可する |
| FR-7b | リスクレベル中で検証 FAIL 時に人間エスカレーションする |
| FR-8 | リスクレベル低でゲート FAIL 時もエスカレーション不要とする |
| FR-8a | リスクレベル低で検証もエスカレーション不要とする |
| FR-9 | 改ざんゲート FAIL はリスクレベル非依存で常時エスカレーションする |
| FR-10 | RFCテンプレートからステータス行を削除する |
| FR-10a | RFCの承認状態をPRマージ状態で判定する |
| FR-10b | RFCレビューコマンドからステータス更新処理を削除する |
| FR-10c | 実装コマンドの前提確認をPRマージ済み確認に変更する |
| FR-11 | RFCゲート基準に仕様書整合性ゲートを新設する |
| FR-11a | 仕様書とRFCのテスト仕様の整合性を検証する |
| FR-11b | 適用条件を仕様書存在かつRFC1件のみとする |
| FR-12 | RFCレビューのキーワードスキャンは新ゲート対応追加不要とする |
| FR-13 | 単一RFC時に全体E2Eテストをスキップする |
| FR-14 | 修正・簡易修正コマンドは変更しない |

### やらないこと

- 修正コマンド・簡易修正コマンドの変更
- 既存のゲートキーパー人格定義の変更
- レビューコマンド・修正コマンド本体のロジック変更
  （呼出元の変更のみ）

## 3. 非機能要件

### 達成すべき要件

| ID | 分類 | 要件 |
|----|------|------|
| NFR-1 | 冪等性 | 任意のフェーズから再開可能であること |
| NFR-2 | 可観測性 | フェーズ判定結果が人間に可視であること |
| NFR-3 | 安全性 | 判定不能時に安全側（再レビュー側）に倒すこと |
| NFR-4 | 保守性 | コマンドネスト深度を最大1に制限すること |
| NFR-5 | 互換性 | 既存の単体コマンド呼出は影響を受けないこと |

### やらないこと

- 並列実行による高速化
- 自動ロールバック機構の導入

## 4. 実現方式

<!-- 機能要件・非機能要件をどう実現するかの高レベル方針。 -->

| 要件 | 実現方式 |
|------|----------|
| FR-1 | 自動RFCコマンド定義と自動実装コマンド定義を削除する |
| FR-2, FR-3, FR-3a | オーケストレータから各コマンドを直接呼出す構造に変更する |
| FR-4〜FR-4f | 統合開発コマンド定義内でシェルコマンドによる状態チェックを行い、フェーズマップ JSON を生成する |
| FR-5〜FR-5b | サービス仕様書テンプレートに §0.5 リスクレベルを追加し、仕様書作成コマンドに引数を追加する |
| FR-6〜FR-9 | オーケストレータ内でリスクレベルに応じたマージ・エスカレーション分岐を実装する |
| FR-10〜FR-10c | テンプレート・レビューコマンド・実装コマンドからステータス関連処理を削除・変更する |
| FR-11〜FR-11b | RFCゲート基準定義に GATE-6 を追加する |
| FR-12 | キーワードスキャン対応表への追加は不要（意味的判定に依存） |
| FR-13 | オーケストレータ内でRFC件数を判定し条件分岐する |
| NFR-1〜NFR-3 | フェーズ判定ロジックを終端逆順チェック・安全側倒しで実装する |
| NFR-4 | 統合開発コマンド→オーケストレータ→各コマンドの2層構造に限定する |

## 5. 代替案の検討

<!-- §4の実現方式に対する代替案。最低2案を比較。 -->

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: フェーズ判定をコマンド定義側で実行 | エージェント側でシェルコマンドにより機械的に状態を判定し、JSON を生成してオーケストレータに渡す | **採用** | フェーズ判定結果を人間が承認できる。オーケストレータは判定不要で単純化される |
| B: フェーズ判定をオーケストレータ内で実行 | オーケストレータが自身で状態を判定し、各コマンドを呼出す | 却下 | フェーズ判定の誤りを人間が事前に検知できない。現行の問題と同じ構造になる |
| C: ステータス行を維持しPRマージ状態と二重管理 | RFCファイル内のステータスとPRマージ状態の両方で管理する | 却下 | 二重管理は不整合の原因。PRマージ状態を唯一の真実とする方が単純で堅牢である |
| D: ステータス行を廃止しPRマージ状態のみで管理 | RFCの承認状態をPRマージ状態のみで判定する | **採用** | 単一の真実源を維持でき、ステータス更新漏れのリスクを排除できる |

## 6. 外部仕様 (External Specification)

<!-- ユーザ・運用者視点の振る舞い。箇条書き＋コード例。 -->

- 自動RFCコマンド・自動実装コマンドは廃止される
  - 実行するとファイル不在エラーとなる
- 統合開発コマンドが全自動導線の唯一の起点となる
- 統合開発コマンド実行時にフェーズテーブルが表示される
  - 各 slug の現在フェーズと開始コマンドが一覧表示される
  - ユーザが承認してからオーケストレータが起動する
- 仕様書作成コマンドに `--risk high|medium|low` 引数が追加される
  - 引数省略時はユーザに確認ダイアログが表示される
- リスクレベルに応じてマージ・エスカレーション動作が変わる
  - 高: ゲート FAIL 時の特定ゲートで人間エスカレーション、PRマージ・検証完了時に人間承認必須
  - 中: ゲート FAIL 時の特定ゲートで人間エスカレーション、ゲート通過時は自動マージ、検証 FAIL 時に人間エスカレーション
  - 低: エスカレーション不要、自動マージ
  - 改ざんゲート FAIL は全リスクレベルで人間エスカレーション
- RFC テンプレートからステータス行が消える
  - RFC レビュー PASS 後の PR Ready 化のみ行われる
- RFC ゲートレビューに GATE-6（仕様書整合性検証）が追加される
  - サービス仕様書が存在し RFC が1件のみの場合に実行される
- 単一 RFC の場合、全体 E2E テストはスキップされる

### 11.9 オーケストレータスクリプトの刷新

- `bin/adev.sh` を以下の構造に刷新する:

```bash
#!/usr/bin/env bash
set -euo pipefail
unset CLAUDECODE

VDEV_HOME="$HOME/projects/vdev"
source "$VDEV_HOME/bin/lib/git-utils.sh"

# --- 引数チェック ---
SPEC_PATH="$1"
PHASE_MAP_FILE="$2"

RISK_LEVEL=$(jq -r '.risk_level' "$PHASE_MAP_FILE")
SLUGS_JSON=$(jq -c '.slugs' "$PHASE_MAP_FILE")
REPO_ROOT="$(git rev-parse --show-toplevel)"
SPEC_DIR="$(dirname "$SPEC_PATH")"
DECISION_LOG="${SPEC_DIR}/adev-decisions.md"

# --- slug ループ ---
echo "$SLUGS_JSON" | jq -c '.[]' | while read -r entry; do
  slug=$(echo "$entry" | jq -r '.slug')
  phase=$(echo "$entry" | jq -r '.phase')

  case "$phase" in
    DONE)
      echo "[${slug}] DONE。スキップ。"
      continue
      ;;
    RFC)
      # /rfc 実行
      # ... claude -p で直接 /rfc を呼出す
      phase="RRFC"  # 次フェーズへフォールスルー
      ;;&
    RRFC)
      # /rrfc + /urfc レビューループ
      # ... claude -p で直接 /rrfc, /urfc を呼出す
      phase="MERGE_RFC"
      ;;&
    MERGE_RFC)
      # RFC PR マージ
      # リスクレベルに応じた分岐
      if [ "$RISK_LEVEL" = "high" ]; then
        # 人間承認待ち → エスカレーション
        echo "ESCALATION_REQUIRED"
        echo "ブロッカー: RFC PR マージに人間承認が必要"
        echo "理由: リスクレベル高"
        echo "推奨アクション: PR を確認しマージしてください"
        echo "再開条件: PR がマージされていること"
        exit 2
      fi
      # medium/low: 自動マージ
      gh pr merge --squash --delete-branch
      phase="IMP"
      ;;&
    IMP)
      # /imp 実行
      # ... claude -p で直接 /imp を呼出す
      phase="RIMP"
      ;;&
    RIMP)
      # /rimp レビューループ
      # ゲート FAIL 時のエスカレーション判定
      # GATE-I0: 全リスクレベルでエスカレーション
      # GATE-I1, GATE-I2:
      #   high/medium → エスカレーション
      #   low → エスカレーション不要
      phase="VFY"
      ;;&
    VFY)
      # /vfy + /rvfy 実行
      # リスクレベルに応じた分岐
      # high: 各イテレーション完了時に人間承認
      # medium: FAIL 時に人間エスカレーション
      # low: エスカレーション不要
      phase="MERGE_IMPL"
      ;;&
    MERGE_IMPL)
      # 実装 PR マージ
      if [ "$RISK_LEVEL" = "high" ]; then
        echo "ESCALATION_REQUIRED"
        echo "ブロッカー: 実装 PR マージに人間承認が必要"
        echo "理由: リスクレベル高"
        echo "推奨アクション: PR を確認しマージしてください"
        echo "再開条件: PR がマージされていること"
        exit 2
      fi
      gh pr merge --squash --delete-branch
      ;;
  esac
done

# --- 全体 E2E テスト ---
SLUG_COUNT=$(echo "$SLUGS_JSON" \
  | jq '[.[] | select(.phase != "DONE")] | length')

if [ "$SLUG_COUNT" -ge 2 ]; then
  echo "=== 全体 E2E テスト ==="
  # /vfy + /rvfy を仕様書パスで実行
else
  echo "=== 単一 RFC のため全体 E2E テストをスキップ ==="
fi
```

上記は構造を示す擬似コードである。各 `claude -p` 呼出は現行の `run_claude_with_recovery` ラッパーを使用する。

### 11.10 リスクレベルによるエスカレーション分岐の詳細

オーケストレータ内の分岐ロジック:

```bash
# ゲート FAIL 時のエスカレーション判定関数
should_escalate_gate_fail() {
  local risk="$1"
  local gate="$2"

  # GATE-0, GATE-I0 は常時エスカレーション
  case "$gate" in
    GATE-0|GATE-I0)
      return 0  # エスカレーション必要
      ;;
  esac

  case "$risk" in
    high|medium)
      # GATE-5, GATE-2, GATE-I1, GATE-I2 でエスカレーション
      case "$gate" in
        GATE-5|GATE-2|GATE-I1|GATE-I2)
          return 0
          ;;
      esac
      ;;
    low)
      return 1  # エスカレーション不要
      ;;
  esac
  return 1
}

# PR マージ前の人間承認判定関数
should_require_merge_approval() {
  local risk="$1"
  case "$risk" in
    high)  return 0 ;;  # 承認必要
    *)     return 1 ;;  # 自動マージ OK
  esac
}

# 検証結果のエスカレーション判定関数
should_escalate_vfy() {
  local risk="$1"
  local result="$2"

  case "$risk" in
    high)
      return 0  # 結果に関わらずエスカレーション
      ;;
    medium)
      [ "$result" = "FAIL" ] && return 0
      return 1
      ;;
    low)
      return 1  # エスカレーション不要
      ;;
  esac
}
```
