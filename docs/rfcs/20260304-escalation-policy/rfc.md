## 1. 背景・動機 (Motivation)

brillat プロジェクトの自動開発実行時に、AI エージェントが人間の介入が必要な場面で独断処理を行う問題が発生した。具体的には、E2E テストでの物理操作依頼や認証情報取得が必要な場面で、ダミー値設定やテストスキップにより作業を継続した。根本原因は、vdev フロー全体に「いつ停止すべきか」「どう伝えるべきか」を定めたエスカレーションルールが存在しないことである。加えて、自動開発の非対話アーキテクチャがエスカレーション手段を持たないため、子プロセスが問題を検知しても人間に伝達する経路がない。本 RFC では3層エスカレーション機構を導入し、行動原則・伝達手段・監視処理の各層で問題を解決する。

## 2. 機能要件

### 達成すべき要件

| ID | 要件 |
|----|------|
| FR-1 | エスカレーションポリシーをルールファイルとして新規作成する |
|       | - 全コマンド・全対話に適用する行動原則として定義 |
|       | - 回避行動の明示的禁止 |
|       | - 即時停止の原則 |
|       | - 3カテゴリのエスカレーション対象定義 |
| FR-2 | 非対話環境でのエスカレーション伝達手段を実装する |
|       | - 子プロセスが構造化マーカーを標準出力に出力する |
|       | - リカバリ関数がマーカーを検出し専用終了コードで終了する |
|       | - 自動開発オーケストレータの各プロンプトにマーカー出力指示を追記する |
| FR-3 | 監視エージェントにエスカレーション検知処理を追加する |
|       | - 専用終了コード検知時にログからエスカレーション内容を抽出しユーザに提示 |
|       | - ユーザの対応完了を待機し、オーケストレータを再起動する |
| FR-4 | コマンド実行ポリシーに自力完遂可能性の制約を追記する |
|       | - 「全ステップは明示的指示」の前提として「AI が自力で完遂可能」を明記 |
| FR-5 | 検証コマンドにエスカレーション判定ステップを追加する |
|       | - 各テスト項目の実行前に AI 単独完遂可能性を判定する |
|       | - 不可能と判定した場合はテスト実行に入らずエスカレーションする |
| FR-6 | 自動実装コマンドのリトライ上限到達報告を改善する |
|       | - 全試行の概要を含める |
|       | - 根本原因分析を含める |
|       | - 推奨アクションを含める |

### やらないこと

- エスカレーション内容の自動解決（人間の介入を代替する仕組み）
- エスカレーション履歴の永続的な管理・追跡
- 対話モードでの自動停止（対話モードでは既存の質問機能で対応可能）

## 3. 非機能要件

### 達成すべき要件

| ID | 分類 | 要件 |
|----|------|------|
| NFR-1 | 安全性 | エスカレーション検知から停止までの間に後続ステップが実行されない |
| NFR-2 | 互換性 | エスカレーション不要な通常実行フローに影響を与えない |
| NFR-3 | 冪等性 | オーケストレータ再起動時に完了済みの slug はスキップされる |
| NFR-4 | 可観測性 | エスカレーション理由が人間にとって十分理解可能な形式で報告される |

### やらないこと

- エスカレーション専用の通知チャネル（Slack、メール等）の構築
- エスカレーション発生頻度のメトリクス収集

## 4. 実現方式

| 要件 | 実現方式 |
|------|----------|
| FR-1: エスカレーションポリシー | ルールファイルとして行動原則を定義する |
|                                | エスカレーション対象を3カテゴリに分類する |
|                                | 即時停止と回避行動禁止を明文化する |
|                                | 報告形式を既存のコミュニケーション原則に準拠させる |
| FR-2: 非対話環境の伝達手段 | 構造化マーカーを標準出力に出力して子プロセスを終了する |
|                            | リカバリ関数でマーカーパターンを検出し終了コード2で終了する |
|                            | オーケストレータの各プロンプトにマーカー出力形式の指示を追記する |
| FR-3: 監視エージェントの検知処理 | 自動開発コマンド定義のモニタリングループにエスカレーション検知を追加する |
|                                  | ログからエスカレーション内容を抽出してユーザに提示する |
|                                  | ユーザ対応後にオーケストレータを再起動する |
| FR-4: コマンド実行ポリシー改修 | 原則セクションに自力完遂制約を1文追記する |
| FR-5: 検証コマンド改修 | 検証実行ステップの冒頭にエスカレーション判定を挿入する |
| FR-6: 自動実装コマンド改修 | リトライ上限到達時の報告テンプレートを拡充する |

## 5. 代替案の検討

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: 3層エスカレーション（採用案） | ルールファイル・シェル伝達・監視検知の3層で多層防御する | **採用** | 対話/非対話の両環境をカバーし、既存アーキテクチャの最小変更で実現できる。各層が独立しており段階的導入が可能 |
| B: 専用エスカレーションサーバ | WebSocket やファイル監視による専用通信チャネルを構築する | 却下 | インフラの複雑化が大きく、現行の非対話アーキテクチャとの統合コストが高い。stdout + 終了コードで十分な情報量を伝達できる |
| C: 対話モードへの切り替え | 非対話子プロセスを対話モードに変更しエスカレーション時に質問する | 却下 | オーケストレータのプロセス管理アーキテクチャの根本的な変更が必要。OOM 防止のための子プロセス分離設計と矛盾する |

## 6. 外部仕様 (External Specification)

- 全コマンド・全対話で AI がエスカレーション対象を検知した場合に即時停止する
  - 物理操作が必要な場合（外部 UI 操作、デバイス確認等）
  - AI が取得不可能な情報が必要な場合（認証情報、シークレット等）
  - 人間の意思決定が必要な場合（要件取捨選択、曖昧な仕様の解釈等）
- エスカレーション時の報告にはブロッカー・理由・推奨アクション・再開条件が含まれる
- 自動開発実行中にエスカレーションが発生した場合:
  - オーケストレータが自動停止する
  - 監視エージェントがエスカレーション内容をユーザに提示する
  - ユーザが対応完了後、オーケストレータが再起動し完了済み作業をスキップして継続する
- 検証コマンドの各テスト項目実行前に AI 単独完遂可能性が判定される
  - 不可能な項目はテスト実行に入らずエスカレーションされる
- 自動実装コマンドのリトライ上限到達時の報告が詳細化される
  - 全試行の概要、根本原因分析、推奨アクションが含まれる

### ロールバック基準と手順

- ロールバック基準: エスカレーション機構の導入により通常の自動開発フローが破壊された場合
- ロールバック手順: git revert で該当コミット群を打ち消す。変更はルールファイル・コマンド定義・ユーティリティに分散しているが、各変更は追記または独立ファイルの新規作成であるため、打ち消しによる副作用は限定的

## 10. 前提条件・依存関係

| 種別 | 内容 |
|------|------|
| 前提 | 非対話子プロセスの出力チャネルは stdout + 終了コードのみである |
| 前提 | オーケストレータの冪等性（完了済み slug スキップ）が実装済みである |
| 前提 | 終了コード2は現行のリカバリ関数で未使用である |
| 依存 | コミュニケーション原則の推奨先行の原則が定義済みであること |

## 11. 詳細設計 (Detailed Design)

### 層1: エスカレーションポリシー（ルールファイル）

`adapters/claude/rules/escalation-policy.md` を新規作成する。

```markdown
# エスカレーションポリシー

## 適用範囲

全コマンド・全対話に適用する。
フェーズ非依存であり、仕様策定・RFC・実装・検証の
すべてに適用される。

## 原則

AI が自力で完遂できないタスクに遭遇した場合、
即座に作業を停止しエスカレーションする。

### 即時停止の原則

エスカレーション対象を検知した時点で
現在のステップを中断する。
後続ステップ・残り項目の処理を禁止する。

### 回避行動の禁止

以下の行動を明示的に禁止する:
- ダミー値・プレースホルダーの設定
- ステップのスキップ
- 仮完了の報告
- テスト項目の省略・簡略化

## エスカレーション対象

### カテゴリ1: 人間の物理操作が必要

- 外部 UI の操作（管理画面、設定画面等）
- 物理デバイスの確認・操作
- 外部サービスへの手動ログイン

### カテゴリ2: AI が取得不可能な情報が必要

- 認証情報（API キー、シークレット、トークン）
- 外部管理画面でのみ確認できる値
- 人間のみがアクセス可能なシステムの情報

### カテゴリ3: 人間の意思決定が必要

- ビジネス上の優先度判断
- 要件の取捨選択
- 曖昧な仕様の解釈
- トレードオフを伴う設計判断

## 報告形式

communication.md の推奨先行の原則に従い、
以下の4項目を報告する:

1. **ブロッカー**: 何が作業を阻んでいるか
2. **理由**: なぜ AI が自力で解決できないか
3. **推奨アクション**: 人間に何をしてほしいか
4. **再開条件**: 何が解決されれば作業を再開できるか
```

### 層2: 非対話環境でのエスカレーション伝達

`bin/lib/git-utils.sh` の `run_claude_with_recovery` 関数に構造化マーカー検出を追加する。

```bash
run_claude_with_recovery() {
  local max_attempts=3
  local attempt=1
  local tmpfile errfile outfile
  tmpfile=$(mktemp)
  errfile=$(mktemp)
  outfile=$(mktemp)
  trap 'rm -f "$tmpfile" "$errfile" "$outfile"' RETURN

  cat > "$tmpfile"

  while [ "$attempt" -le "$max_attempts" ]; do
    echo "[claude] 試行 ${attempt}/${max_attempts}" >&2

    if [ "$attempt" -eq 1 ]; then
      if cat "$tmpfile" \
        | claude -p "$@" 2>"$errfile" | tee "$outfile"; then
        # エスカレーションマーカー検出
        if grep -q "ESCALATION_REQUIRED" "$outfile"; then
          echo "[claude] エスカレーション検出" >&2
          cat "$outfile"
          return 2
        fi
        return 0
      fi
    else
      local prev_error
      prev_error=$(cat "$errfile")
      local orig_prompt
      orig_prompt=$(cat "$tmpfile")
      if {
        cat <<RECOVERY_EOF
前回の実行が以下のエラーで失敗した。
エラー内容を分析し、問題を調査・修復した上で、
元のタスクを完遂せよ。

--- エラー出力 ---
${prev_error}

--- 元のプロンプト ---
${orig_prompt}
RECOVERY_EOF
      } | claude -p "$@" 2>"$errfile" | tee "$outfile"; then
        if grep -q "ESCALATION_REQUIRED" "$outfile"; then
          echo "[claude] エスカレーション検出" >&2
          cat "$outfile"
          return 2
        fi
        return 0
      fi
    fi

    echo "[claude] 試行 ${attempt} 失敗" >&2
    attempt=$((attempt + 1))
  done

  echo "[claude] 最大試行回数(${max_attempts})に到達" >&2
  return 1
}
```

子プロセスが出力する構造化マーカーの形式は以下の通りである。

```
ESCALATION_REQUIRED
ブロッカー: {ブロッカーの内容}
理由: {AI が自力で解決できない理由}
推奨アクション: {人間に求めるアクション}
再開条件: {再開に必要な条件}
```
