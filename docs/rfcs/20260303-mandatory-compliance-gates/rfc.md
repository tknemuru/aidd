## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

自動開発において、仕様書で明記された技術的意思決定が RFC・実装の各工程に伝達されず、実装エージェントがテンプレート代替で LLM 連携を先送りする事故が発生した。根本原因は3つある。仕様書の重要な技術的意思決定が後続作業者に伝わらない構造、ゲートキーパーが仕様書を参照せず事実を捏造して PASS させた検証不備、および実装が RFC から逸脱したことを検出する仕組みの欠如である。本 RFC では「絶対守ること」セクションによる重要事項の明示的伝達、ゲートキーパーの根拠引用義務化、および実装適合ゲートレビューの新設により、仕様→設計→実装の一貫性を構造的に保証する。

## 5. 代替案の検討

<!-- §4の実現方式に対する代替案。最低2案を比較。 -->

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: ゲートキーパー方式 | キーワードスキャン＋単一ゲートキーパーによる実装適合検証 | **採用** | RFC ゲートレビューと同一構造で保守性が高い |

| C: テストのみ方式 | 実装ゲートレビューを設けず検証コマンドのみで担保 | 却下 | 検証コマンドは E2E テストの実行のみで RFC 適合性を検証しない |

## 6. 外部仕様 (External Specification)

- 仕様書テンプレートの先頭に必須遵守事項セクションが追加される
  - 人間のみが記述するセクションであり、AI は空欄のまま残す
- RFC テンプレートの先頭に必須遵守事項セクションが追加される
  - 人間のみが記述するセクションであり、AI は空欄のまま残す
- 自動開発実行時、仕様書の必須遵守事項が RFC に自動転記される
- RFC レビューに必須遵守事項違反排除ゲートが追加される
  - 必須遵守事項が空の場合は自動 PASS
- 実装ゲートレビューコマンドが新規追加される
  - 3つのゲート: 必須遵守事項違反排除、先送り排除、即時中断強制
  - 必須遵守事項が空の場合、最初のゲートは自動 PASS
- 自動実装コマンドに実装ゲートレビューが組み込まれる
  - FAIL 時は最大5回の修正ループ
  - 5回到達で FAIL の場合はユーザに報告して停止
- ゲートキーパーが判定根拠を述べる際、原文の具体的記述を引用する義務がある
  - 引用元が実在しない主張は証拠として認めない

### 11.3 自動開発スクリプトでの §0 機械的コピー

`bin/adev.sh` の Step 3-1（自動 RFC 実行）完了後、Step 3-2（RFC PR マージ）の前に §0 コピー処理を挿入する。

```bash
# --- Step 3-1.5: §0 機械的コピー ---
SPEC_SECTION0=$(sed -n '/^## 0\. 絶対守ること/,/^## 1\./{ /^## 1\./d; p; }' \
  "$SPEC_PATH")
if [ -n "$SPEC_SECTION0" ]; then
  RFC_FILE="$REPO_ROOT/docs/rfcs/${slug}/rfc.md"
  if [ -f "$RFC_FILE" ]; then
    # RFC の §0 コメント行を含むセクションを仕様書の §0 で置換
    sed -i '/^## 0\. 絶対守ること/,/^## 1\./{
      /^## 1\./!d
    }' "$RFC_FILE"
    # §1 の直前に仕様書の §0 を挿入
    sed -i "/^## 1\./i\\
${SPEC_SECTION0}" "$RFC_FILE"
    # 変更をコミット
    git -C "$REPO_ROOT" add "$RFC_FILE"
    git -C "$REPO_ROOT" commit -m \
      "docs: copy §0 from service spec to RFC for ${slug}" || true
    git -C "$REPO_ROOT" push || true
    log_decision "$slug" "§0コピー" "sed注入" "成功" "-"
  fi
fi
```

改行を含む sed 置換のため、一時ファイルを使う方式に切り替える可能性がある。

### 11.4 GATE-5 の追加

`prompts/criterias/gate-criteria-rfc.md` に以下を追記する。

```markdown
## GATE-5: 絶対守ることの違反排除

※ §0 が空欄の場合、本ゲートは自動 PASS とする。

### 基準 A（適合証明）
- §0 の各項目に対して、RFC の設計が適合している
  積極的証拠があるか

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- §0 の各項目に違反する設計がないか
```

`prompts/criterias/gate-criteria-impl.md` を以下の内容で新規作成する。

```markdown
# 実装ゲートキーパー検証項目

## GATE-I1: 絶対守ることの違反排除

※ §0 が空欄の場合、本ゲートは自動 PASS とする。

### 基準 A（適合証明）
- §0 の各項目に対して、実装が適合している
  積極的証拠があるか
- 証拠は実装コードのファイルパス・関数名・
  行番号を引用して示すこと

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- §0 の項目に違反する実装がないか
- スタブ・テンプレート・固定値での代替がないか
- 「将来実装」「TODO」としての先送りがないか

## GATE-I2: 勝手な先送り排除

### 基準 A（適合証明）
- RFC の機能要件（§2）の全項目に対応する
  実装が存在するか

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- 「将来実装」「TODO」「暫定」コメントが
  実装コード内に存在しないか
- RFC の実現方式（§4）と異なる簡略化された
  代替実装がないか

## GATE-I3: フォールバック排除（Fail-Fast 強制）

### 基準 A（適合証明）
- すべてのエラー・異常状態に対し、即座に処理を
  中断しエラーを呼び出し元に伝播する実装であるか
- 入力値・前提条件の検証が処理の先頭で行われ、
  違反時に即座にエラーとなる実装であるか

### 基準 B（不適合排除）
- キーワードスキャン検出 → 基準 B 違反確定
- エラー発生時にデフォルト値で処理を続行する
  実装がないか
- 例外を catch して握りつぶす実装がないか
- 取得失敗時に空配列・空文字列・ゼロ値等で
  代替する実装がないか
- 条件分岐で「それ以外」を安全側に倒す
  実装がないか
```

### 11.8 ゲートキーパー人格定義への根拠引用義務追加

`prompts/roles/gatekeeper.md` に以下のセクションを追加する。

```markdown
## 根拠引用義務

基準 A の適合証拠を述べる際、以下を遵守せよ:

- RFC 本文・仕様書・実装コードの具体的な記述を引用すること
  - セクション番号（例: §2 の F-1）
  - ファイルパス:行番号（例: src/main.rs:42）
  - 該当文言の直接引用
- 引用元が実在しない主張は証拠として認めない
- 「〜の前提として設定済みである」
  「〜は既知である」等の外部知識に基づく推定は禁止
- 文書に記載がなければ「証拠なし」として
  基準 A 不充足 = FAIL とする
```

### 単体テスト仕様

| テスト対象 | 検証観点 |
|-----------|----------|
| §0 テンプレートコメント | 仕様書・RFC 両テンプレートにコメントが存在すること |
| §0 sed 抽出処理 | 仕様書 §0 の内容が正しく抽出されること |
| §0 sed 抽出処理（空欄） | §0 が空の場合に空文字列が返ること |
| §0 sed 注入処理 | RFC の §0 が仕様書の内容で正しく置換されること |
| GATE-5 空欄判定 | §0 空欄時に GATE-5 が自動 PASS となること |
| GATE-I1 空欄判定 | §0 空欄時に GATE-I1 が自動 PASS となること |
| 実装キーワードスキャン | 禁止用語が実装コード内で正しく検出されること |
| 実装キーワードスキャン（検出なし） | 禁止用語がない場合に「検出なし」が返ること |
| 修正ループ上限 | impl_gate_attempts が5に達した時に停止すること |
