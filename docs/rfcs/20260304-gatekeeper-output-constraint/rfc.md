## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

ゲートキーパーレビューシステムに構造的欠陥がある。FAIL 時に修正方法を自由生成するため表面的な修正推奨が発生し、キーワードスキャン結果を LLM に渡した上で出力抑制を指示しているため検出事実が漏洩する。これらは「LLM に情報を与えた上で出力に含めるなと指示する」アーキテクチャに起因し、プロンプト修正では原理的に解決できない。本 RFC はゲートキーパーの出力形式を NG 引用+固定フィードバック文に制約し、キーワードスキャン結果をゲートキーパーに渡さない方式へ移行することで、これらの問題を構造的に解決する。

## 6. 外部仕様 (External Specification)

- FAIL 時のレビュー結果ファイルの出力形式が変わる
  - 変更前: 各ゲートの FAIL 理由に修正提案を含む
  - 変更後: NG 箇所の引用と固定フィードバック文のみ
- キーワード検出による FAIL が
  ゲートキーパー Task 内ではなく
  呼び出し元コマンドの後処理で適用される
- ユーザがレビュー結果を見たとき、
  修正方法ではなく「何が NG で何を満たすべきか」が
  示される

## 11. 詳細設計 (Detailed Design)

### ゲートキーパー人格定義の変更

- キーワードスキャン検出時のルールセクションを全削除する
- フィードバック出力規則を以下に書き換える

```markdown
## フィードバック出力規則

- FAIL 時の出力は以下の2要素のみとせよ:
  - NG 箇所の引用（セクション番号・行番号・該当文言）
  - 検証項目に定義された固定フィードバック文の転記
- 以下の出力を一切禁止する:
  - 修正方法の提案
  - 改善案の示唆
  - 代替アプローチの提示
  - 修正の具体的な記述追加の指示
```

### RFC ゲート検証項目の変更

- 全ゲートの基準 B から以下の行を削除する

```
- キーワードスキャン検出 → 基準 B 違反確定
```

- 各ゲートに以下の形式で固定フィードバック文を追加する

GATE-1 の固定フィードバック文の例を示す。

```markdown
### FAIL 時の固定フィードバック

「全テストケースが以下を満たす状態にせよ:
外部サービス実接続、
ドメイン固有の実在データ使用、
ユーザ操作起点→最終アウトカム確認の構造、
各機能要件に正常系・異常系/境界値の2ケース以上。」
```

各ゲートの固定フィードバック文は以下の通りである。

- GATE-1: 「全テストケースが以下を満たす状態にせよ:
  外部サービス実接続、
  ドメイン固有の実在データ使用、
  ユーザ操作起点→最終アウトカム確認の構造、
  各機能要件に正常系・異常系/境界値の2ケース以上。」
- GATE-2: 「勝手な先送りは許可しない。
  全機能要件を当 RFC のスコープとして
  設計・Task 計画に含めよ。」
- GATE-3: 「設計上必要な環境状態変更
  （DDL、cron、デプロイ設定、
  システムサービス設定、環境変数設定等）を
  全て Task 計画に含めよ。」
- GATE-4: 「全てのエラー・異常状態で即座に処理を中断し
  エラーを呼び出し元に伝播する設計にせよ。
  デフォルト値での続行・例外の握り潰し・
  代替値での処理継続を全て排除せよ。」
- GATE-5: 「§0 に定義された全項目に
  適合する設計にせよ。」

### 実装ゲート検証項目の変更

- 全ゲートの基準 B から以下の行を削除する

```
- キーワードスキャン検出 → 基準 B 違反確定
```

- 各ゲートに固定フィードバック文を追加する
  - GATE-I1: 「§0 に定義された全項目に
    適合する実装にせよ。」
  - GATE-I2: 「全ての RFC §2 要件に対応する
    実装を完了せよ。
    TODO・FIXME・暫定・仮実装を全て排除せよ。」
  - GATE-I3: 「全てのエラー・異常状態で即座に
    処理を中断しエラーを呼び出し元に伝播する
    実装にせよ。デフォルト値での続行・
    例外の握り潰し・代替値での処理継続を
    全て排除せよ。」

### レビューコマンドの Task プロンプト変更

RFC レビューコマンド・実装レビューコマンド共通で以下を変更する。

- Task プロンプトから以下を削除する:
  - 「キーワードスキャン結果」セクション
  - 「キーワードスキャンで検出がある場合、
    該当ゲートは基準 B 違反確定として
    無条件 FAIL せよ。」の指示行
  - 「禁止語の検出事実を指摘に含めるな。」の指示行
- Task プロンプトの出力制約を以下に変更する:
  - 「FAIL の場合、PASS に必要な具体的修正を記載せよ。」
    を削除する
  - 代わりに「FAIL の場合、NG 箇所の引用と
    固定フィードバック文の転記のみを記載せよ。」
    を追加する

### Step 5.5: キーワード→ゲート対応表による機械的 FAIL 上書き

Step 5（ゲートキーパー検証実行）の後に Step 5.5 を追加する。

```markdown
### Step 5.5: キーワード検出による FAIL 上書き

Step 4.5 でキーワードが検出された場合、
以下のキーワード→ゲート対応表に基づき、
該当ゲートの判定を機械的に FAIL に上書きせよ。

上書き時のフィードバックは、
検証項目に定義された該当ゲートの
固定フィードバック文を使用する。

ゲートキーパーが当該ゲートを既に FAIL 判定
している場合も、フィードバック文を
固定フィードバック文で上書きする。
```
