## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

gale プロジェクトにおいて、承認済み RFC が実装 PR 内で AI により無断改変されていたことが判明した。根本原因はワークフロー定義の Living Document ルールが「承認済み RFC の変更は正当」と解釈される根拠を AI に与えていたことである。さらに、RFC レビュー・実装レビュー・検証の各コマンドに仕様書の改ざんチェックが存在せず、改変の検出もできなかった。承認済み仕様の不変性を制度として保証し、機械的な差分検出で改ざんを排除する仕組みが必要である。

## 2. 機能要件

### 達成すべき要件

<!-- コード識別子禁止。機能名称またはドメイン用語で記述。 -->

| ID | 要件 |
|----|------|
| F-1 | ワークフロー定義の Living Document ルールを廃止し、 |
|     | Immutable Specification ルールに置換する |
|     | （承認済み RFC およびマージ済みサービス仕様書の |
|     | 内容変更を禁止し、乖離発生時は |
|     | 即時停止・人間報告・承認取得を義務付ける） |
| F-2 | RFC レビューコマンドに |
|     | 仕様書改ざん排除ゲートを先頭に追加する |
|     | （RFC ブランチ上でサービス仕様書が |
|     | デフォルトブランチから変更されていないことを検証） |
| F-3 | 実装レビューコマンドに |
|     | 仕様改ざん排除ゲートを先頭に追加する |
|     | （feature ブランチ上で承認済み RFC と |
|     | サービス仕様書がデフォルトブランチから |
|     | 変更されていないことを検証） |
| F-4 | 検証コマンドに仕様改ざんチェックを追加する |
|     | （RFC モード: 承認済み RFC の差分検出、 |
|     | service-spec モード: サービス仕様書の差分検出。 |
|     | 差分存在時は即時停止・エスカレーション） |
| F-5 | 実装コマンドに予防的ガードを追加する |
|     | （実装中の RFC およびサービス仕様書の |
|     | 内容変更を明示的に禁止し、 |
|     | 乖離発見時は即時停止・エスカレーション） |

### やらないこと

- 既存ゲート番号体系（GATE-1〜5, GATE-I1〜I3）の変更
- キーワードスキャンの仕組みの変更
- 改ざんチェックにおける AI の裁量判断の導入
- 承認済み RFC のステータスフィールド変更の禁止
  （ステータスはワークフロー上の正当な変更のため対象外）

|       |        | 追加する形式とし、 |
|       |        | 既存番号を変更しないこと |

|       |        | AI による自動修正を禁止し、 |

- 改ざんチェック結果の永続化（レビュー結果ファイルに含まれるため不要）
- 改ざん検出時の自動ロールバック

## 6. 外部仕様 (External Specification)

<!-- ユーザ・運用者視点の振る舞い。箇条書き＋コード例。 -->

- ワークフロー定義から Living Document ルールが廃止され、
  Immutable Specification ルールに置き換わる
- RFC レビュー実行時、サービス仕様書が
  デフォルトブランチから変更されていると GATE-0 FAIL となる
- 実装レビュー実行時、承認済み RFC またはサービス仕様書が
  デフォルトブランチから変更されていると GATE-I0 FAIL となる
- 検証コマンド実行時、仕様ファイルに差分が検出されると
  即時停止しエスカレーションが発生する
- 改ざんゲートの FAIL 時は AI による自動修正が禁止され、
  人間の承認が必須となる
- 実装コマンド実行時、RFC およびサービス仕様書の
  内容変更が明示的に禁止される
- GATE-0 FAIL 時の出力形式:

```
## GATE-0: 仕様書改ざん排除
**判定: FAIL**
- NG: （改ざん検出対象ファイルと差分の要約）
- 対応: （FAIL 時の固定対応文）
```

- GATE-I0 FAIL 時の出力形式:

```
## GATE-I0: 仕様改ざん排除
**判定: FAIL**
- NG: （改ざん検出対象ファイルと差分の要約）
- 対応: （FAIL 時の固定対応文）
```

- デフォルトブランチの検出方式:

```bash
# 第1候補: symbolic-ref
DEFAULT_BRANCH=$(git symbolic-ref refs/remotes/origin/HEAD \
  2>/dev/null | sed 's@^refs/remotes/origin/@@')
# 第2候補: remote show
if [ -z "$DEFAULT_BRANCH" ]; then
  DEFAULT_BRANCH=$(git remote show origin 2>/dev/null \
    | grep 'HEAD branch' | cut -d: -f2 | tr -d ' ')
fi
```

- 差分検出方式:

```bash
# 差分が空なら PASS、非空なら FAIL
git diff origin/<default-branch> -- <file-path>
```

## 11. 詳細設計 (Detailed Design)

<!-- 箇条書き＋コード例。 -->
<!-- コードブロック間補足は2文以内の散文許可。 -->

### 変更1: ワークフロー定義の Living Document ルール置換

`workflow/rfc-driven.md` の69行目を以下に置換する。

変更前:

```markdown
- **Living Document**: 設計は実装中に変化しうる。その場合、コードだけでなく RFC も最新の状態に更新し、差異を発生させないこと。
```

変更後:

```markdown
- **Immutable Specification**: 承認済みRFC（ステータス「Accepted」以降）およびマージ済みサービス仕様書の内容変更を禁止する。実装中に仕様との乖離が発生した場合、AIは即座に停止し、乖離の内容・原因・推奨対応を人間に報告して承認を得ること。
```

### 変更2: RFC ゲート検証項目への GATE-0 追加

`prompts/criterias/gate-criteria-rfc.md` の先頭（GATE-1 の前）に以下を追加する。

```markdown
## GATE-0: 仕様書改ざん排除

### 基準 A（適合証明）
- RFC ブランチ上でサービス仕様書（spec.md）が
  デフォルトブランチから変更されていないか

### 基準 B（不適合排除）
- サービス仕様書に対するいかなる差分も存在しないか

### FAIL 時の固定対応文

仕様書の改ざんを検出。
サービス仕様書がデフォルトブランチから変更されている。
```
