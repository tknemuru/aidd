## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

前回 RFC「ゲートキーパー出力制約」の実運用で3つの問題が発覚した。第一に、キーワード FAIL 上書き時の NG 行にキーワード名・検出箇所・引用文が含まれるため、実装 AI がキーワードをリネームして検出を回避する「回避修正」で通過してしまう。第二に、フォールバック排除ゲートの基準 B がエラーハンドリング層のみを対象としており、本来の処理方式を省略して簡易代替処理で置き換える設計レベルのフォールバックを検出できなかった。第三に、レビューイテレーション上限が各コマンド間でバラバラであり統一されていない。本 RFC はこれら3問題を一括で修正する。

## 6. 外部仕様 (External Specification)

<!-- ユーザ・運用者視点の振る舞い。箇条書き＋コード例。 -->

- FAIL 時のレビュー結果ファイルにおいて、
  キーワード検出による FAIL のフィードバックが
  固定テキストのみとなり、
  キーワード名・検出箇所・引用文が含まれなくなる
- フォールバック排除ゲートが設計レベルの
  フォールバック（本来の処理方式を省略し
  簡易代替処理で置き換える設計）も検出する
- 全レビューコマンドのイテレーション上限が
  8回に統一される
- RFC レビュー・実装レビューのラウンド上限が
  8回となり、超過時はエスカレーション停止する

### RFC ゲート検証項目の固定出力テキスト再設計

GATE-1, GATE-2, GATE-4 の「FAIL 時の固定フィードバック」を「FAIL 時の固定出力」に名称変更し、以下のテキストに置き換える。

GATE-1:

```
NG: 不当なテスト設計を検出。
モック・スタブ・ダミーデータを使った
E2E テストは絶対に許可しない。
テスト設計を根本的に見直せ。
全テストケースが以下を満たす状態にせよ:
外部サービス実接続、
ドメイン固有の実在データ使用、
ユーザ操作起点→最終アウトカム確認の構造、
各機能要件に正常系・異常系/境界値の2ケース以上。
人間へのエスカレーションが必要ならば、
escalation-policy.md に従い、エスカレーションせよ。
```

GATE-2:

```
NG: 不当な先送り設計を検出。
勝手な先送りは絶対に許可しない。
設計から根本的に見直せ。
全機能要件を当 RFC のスコープとして
設計・Task 計画に含めよ。
人間へのエスカレーションが必要ならば、
escalation-policy.md に従い、エスカレーションせよ。
```

GATE-4:

```
NG: 不当なフォールバック設計を検出。
フォールバックは絶対に許可しない。
設計から根本的に見直せ。
全てのエラー・異常状態で即座に処理を中断し
```
