## 1. 背景・動機 (Motivation)

<!-- 散文で記述。ただし5文以内。 -->

WSL2 環境では Windows の Cドライブがマウントポイント経由で参照可能であり、Claude Code から意図せずアクセスされるリスクがある。ファイル読み取り・編集・書き込みツールに加え、シェルコマンド経由のアクセスも防御が必要である。Claude Code の deny ルールには未修正のバグが複数報告されており、deny 単体では信頼性が不十分である。そのため deny ルールとシェルフックによる二重防御を構成し、いずれか一方が機能すればアクセスを確実にブロックする設計とする。

## 2. 機能要件

### 達成すべき要件

<!-- コード識別子禁止。機能名称またはドメイン用語で記述。 -->

| ID | 要件 |
|----|------|
| F1 | ファイル読み取りツールによる Cドライブ配下の全ファイル読み取りを禁止する |
| F2 | ファイル編集ツールによる Cドライブ配下の全ファイル編集を禁止する |
| F3 | ファイル書き込みツールによる Cドライブ配下の全ファイル書き込みを禁止する |
| F4 | シェルコマンド経由での Cドライブへのアクセスを禁止する |
| F5 | deny ルールのバグ回避として、ツール実行前フックで二重防御する |

### やらないこと

- Cドライブ内の特定フォルダを例外的に許可する設定
- Dドライブ等、Cドライブ以外のマウントポイントの防御
- プロジェクトレベル設定への反映（ユーザレベル設定のみ対象）

## 3. 非機能要件

### 達成すべき要件

| ID | 分類 | 要件 |
|----|------|------|
| NF1 | 信頼性 | deny ルールにバグがあっても、フックが独立してブロックする |
| NF2 | 保守性 | 設定は単一ファイルで完結し、構造が自明である |
| NF3 | 互換性 | 既存の Claude Code 設定と競合しない |

### やらないこと

- deny ルールのバグ修正そのものへの対応（上流の責務）

## 4. 実現方式

<!-- 機能要件・非機能要件をどう実現するかの高レベル方針。 -->

| 要件 | 実現方式 |
|------|----------|
| F1-F3 | ユーザ設定の deny リストにツール別パスパターンを登録 |
| F4 | deny リストにシェルツールの Cドライブ参照パターンを登録 |
| F5, NF1 | ツール実行前フックでパス・コマンド文字列を検査しブロック |
| NF2 | ユーザレベル設定ファイル1箇所のみに集約 |

## 5. 代替案の検討

<!-- §4の実現方式に対する代替案。最低2案を比較。 -->

| 案 | 概要 | 採否 | 決定的理由 |
|----|------|------|------------|
| A: deny + フック二重防御 | deny ルールとシェルフックの両方で防御 | **採用** | deny のバグを補償でき、単一障害点がない |
| B: deny ルール単体 | deny ルールのみで防御 | 却下 | 既知バグにより突破される可能性がある |
| C: フック単体 | シェルフックのみで防御 | 却下 | deny が正常動作する場合の早期ブロックを失う |

## 6. 外部仕様 (External Specification)

<!-- ユーザ・運用者視点の振る舞い。箇条書き＋コード例。 -->

- Claude Code がファイルツールで Cドライブ配下を操作しようとした場合:
  - deny ルールにより拒否される
  - deny が突破された場合でもフックがブロックし、理由を出力する
- Claude Code がシェルコマンドで Cドライブにアクセスしようとした場合:
  - deny ルールにより拒否される
  - deny が突破された場合でもフックがブロックし、理由を出力する
- Cドライブ以外（Linux ファイルシステム等）へのアクセスは従来通り制限なし

## 11. 詳細設計 (Detailed Design)

<!-- 箇条書き＋コード例。 -->
<!-- コードブロック間補足は2文以内の散文許可。 -->

- 対象ファイル: `~/.claude/settings.json`
- 設定構造は deny リストとフック定義の2セクションで構成する

deny ルールでは、ファイルツール3種とシェルツールのパターンを定義する。

```json
{
  "permissions": {
    "deny": [
      "Read(//mnt/c/**)",
      "Edit(//mnt/c/**)",
      "Write(//mnt/c/**)",
      "Bash(/mnt/c:*)",
      "Bash(*:/mnt/c*)"
    ]
  }
}
```

- `//mnt/c/**` の `//` はファイルシステムルートからの絶対パスを表す
- シェルツールの deny は2パターンで網羅する:
  - コマンド先頭がマウントポイントで始まるパターン
  - コマンド引数中にマウントポイントを含むパターン

ツール実行前フックでは、ツール入力を検査しブロック判定を行う。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CC_TOOL_INPUT\" | grep -q '/mnt/c'; then echo '{\"decision\": \"block\", \"reason\": \"Access to /mnt/c is blocked for security\"}'; fi"
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CC_TOOL_INPUT\" | grep -q '/mnt/c'; then echo '{\"decision\": \"block\", \"reason\": \"Bash access to /mnt/c is blocked for security\"}'; fi"
          }
        ]
      }
    ]
  }
}
```

- フックは `$CC_TOOL_INPUT` 環境変数でツールへの入力を受け取る
- パス文字列に `/mnt/c` が含まれるか検査する
- 検出時は JSON 形式でブロック判定と理由を標準出力に書き出す
- 非検出時は何も出力せず（暗黙の許可）処理を続行する

最終的な設定ファイルの全体像は以下の通りである。

```json
{
  "permissions": {
    "deny": [
      "Read(//mnt/c/**)",
      "Edit(//mnt/c/**)",
      "Write(//mnt/c/**)",
      "Bash(/mnt/c:*)",
      "Bash(*:/mnt/c*)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CC_TOOL_INPUT\" | grep -q '/mnt/c'; then echo '{\"decision\": \"block\", \"reason\": \"Access to /mnt/c is blocked for security\"}'; fi"
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CC_TOOL_INPUT\" | grep -q '/mnt/c'; then echo '{\"decision\": \"block\", \"reason\": \"Bash access to /mnt/c is blocked for security\"}'; fi"
          }
        ]
      }
    ]
  }
}
```

### 単体テスト仕様

| テスト対象 | 検証観点 |
|-----------|----------|
| deny ルールのパスパターン | Cドライブ配下のパスがマッチすること |
| deny ルールのパスパターン | Cドライブ以外のパスがマッチしないこと |
| フックのファイルツール検査 | ツール入力にマウントポイントを含む場合にブロック JSON を出力すること |
| フックのファイルツール検査 | ツール入力にマウントポイントを含まない場合に何も出力しないこと |
| フックのシェルツール検査 | コマンドにマウントポイントを含む場合にブロック JSON を出力すること |
| フックのシェルツール検査 | コマンドにマウントポイントを含まない場合に何も出力しないこと |
| 設定ファイル全体 | JSON 構文が正しく、パースエラーがないこと |
