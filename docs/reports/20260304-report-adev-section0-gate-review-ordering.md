# adev パイプラインにおける §0 ゲートレビュー順序の構造的欠陥

## 概要

`adev.sh` の自動開発パイプラインにおいて、サービス仕様書の「§0 絶対守ること」が RFC にコピーされるタイミングが GATE-5（絶対守ることの違反排除）レビューの**後**になっており、GATE-5 が RFC レベルで永久に発火しない構造的欠陥が確認された。

## 調査の経緯

`~/projects/abel` の RFC「20260304-overconfidence-diagnosis」において、RFC の §0 に記載があるにもかかわらず、ゲートレビュー R1・R2 の両方で「根拠: §0 が空欄のため自動 PASS」となっていた。サービス仕様書の §0 がゲートレビュー完了後にコピーされたのではないかという疑義が持ち上がった。

## 調査結果

### git 履歴による時系列の確認（abel リポジトリ）

| 時刻 | コミット | 内容 | §0 状態 |
|------|----------|------|---------|
| 16:02 | `17a0b62` | RFC 初稿追加 | 空欄 |
| 16:05 | `d0e966a` | Gate Review R1 作成 → GATE-5 自動 PASS | 空欄 |
| 16:12 | `e7f63e7` | Gate Review R2 作成 → GATE-5 自動 PASS | 空欄 |
| 16:13 | `da0e3ed` | RFC ステータスを Accepted に変更 | 空欄 |
| 16:13 | `b88a5df` | `docs: copy §0 from service spec to RFC` | **コピー済** |

§0 が RFC に書き込まれたのは、両方の Gate Review が完了し RFC が Accepted になった後。

### `adev.sh` のパイプライン実行順序

| 順序 | ステップ | 処理内容 | §0 状態 |
|------|----------|----------|---------|
| Step 3-1 | `/arfc` 実行 | RFC 作成 → `/rrfc` ゲートレビューループ → Accepted | 空欄 |
| Step 3-1.5 | シェルスクリプト | `sed`/`awk` で §0 をサービス仕様書から RFC にコピー | コピー済 |
| Step 3-2 | PR マージ | RFC PR をスカッシュマージ | コピー済 |
| Step 3-3 | `/aimp` 実行 | 実装 → `/rimp` 実装ゲートレビュー（GATE-I1） | コピー済 |

### 根本原因

`adev.sh` L202-224 の §0 コピー（Step 3-1.5）が `/arfc`（Step 3-1）の完了後に配置されている。`/arfc` は単一の Claude コマンドとして Phase 1（RFC 作成）と Phase 2（レビューループ）を連続実行するため、シェルはその中間に介入できない。

結果として:
- **RFC レベルの GATE-5**: §0 は必ず空欄 → 永久に自動 PASS（死んだゲート）
- **実装レベルの GATE-I1**: RFC マージ後に実行されるため §0 は存在 → 正常に機能

RFC の設計が §0 の制約に適合しているかを確認する唯一の機会が、順序の逆転により無効化されている。

## 提案

### 推奨案: `/arfc` に Phase 1.5（§0 注入ステップ）を追加

`arfc.md` の Phase 1（RFC 作成）と Phase 2（レビューループ）の間に、§0 を機械的にコピーするステップを追加する。

`/arfc` は既にサービス仕様書の全文を `$ARGUMENTS` として受け取っている。ここから §0 を抽出して RFC ファイルに転記するのは AI の「執筆」ではなく「機械的コピー」であり、「AI は空欄のまま残す」というルールの趣旨（AI が勝手に制約を定義しない）に反しない。

同時に `adev.sh` の Step 3-1.5（L202-224）を削除し、重複コピーを防止する。

**トレードオフ**:
- `/arfc` 単体でも §0 が正しく注入される（`adev.sh` 経由でなくても動作する）
- `/rfc` の「AI は §0 を空欄に」というルールとの整合性は、`/arfc` のオーケストレータが機械コピーするという役割分担で維持できる
- `/arfc` が元ネタとしてサービス仕様書を受け取らないケース（手動 RFC 等）では §0 は空欄のまま（既存動作と同一）

### 代替案: `adev.sh` で `/arfc` を2段階に分割

`adev.sh` を修正し、RFC 作成とレビューを別々の Claude 呼び出しにする。

```
Step 3-1a: /rfc のみ実行（RFC作成）
Step 3-1.5: シェルで §0 コピー（既存ロジック流用）
Step 3-1b: /rrfc + /urfc ループ実行（レビュー）
```

**トレードオフ**:
- シェル側の制御で順序を保証できる（確実）
- `/arfc` の抽象化が崩れる（`adev.sh` が `/arfc` の内部構造を知る必要がある）
- `/arfc` 単体で使う場合は引き続き §0 が空欄のまま（問題が残る）

## 参考

- Stage-Gate プロセスにおけるゲートの前提条件はゲート実行前に充足されていなければならないのが原則（[Smartsheet](https://www.smartsheet.com/phase-gate-process), [testRigor](https://testrigor.com/blog/software-quality-gates/)）
- RFC 20260303-mandatory-compliance-gates で §0 は「人間が記述し、自動パイプラインで伝播させる単一チャネル」として設計された
