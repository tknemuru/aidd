# 実装ゲートキーパー検証項目

## GATE-I0: 仕様改ざん排除

### 基準 A（適合証明）
- feature ブランチ上で承認済み RFC（rfc.md）が
  デフォルトブランチから変更されていないか
- feature ブランチ上でサービス仕様書（spec.md）が
  デフォルトブランチから変更されていないか

### 基準 B（不適合排除）
- 承認済み RFC に対するいかなる差分も存在しないか
- サービス仕様書に対するいかなる差分も存在しないか

### FAIL 時の固定対応文

仕様の改ざんを検出。
承認済み RFC またはサービス仕様書が
デフォルトブランチから変更されている。
AI による自動修正は禁止。
人間が差分を確認し、対応を判断せよ。

## GATE-I1: 絶対守ることの違反排除

※ §0 が空欄の場合、本ゲートは自動 PASS とする。

### 基準 A（適合証明）
- §0 の各項目に対して、実装が適合している
  積極的証拠があるか
- 証拠は実装コードのファイルパス・関数名・
  行番号を引用して示すこと

### 基準 B（不適合排除）
- §0 の項目に違反する実装がないか

### FAIL 時の固定対応文

§0 に定義された全項目に
適合する実装にせよ。

## GATE-I2: 勝手な先送り排除

### 基準 A（適合証明）
- RFC の機能要件（§2）の全項目に対応する
  実装が存在するか

### 基準 B（不適合排除）
- 「将来実装」「TODO」「暫定」コメントが
  実装コード内に存在しないか
- RFC の実現方式（§4）と異なる簡略化された
  代替実装がないか

### FAIL 時の固定対応文

不当な先送り設計を検出。
勝手な先送りは絶対に許可しない。
設計から根本的に見直せ。

全機能要件を当 RFC のスコープとして
設計・Task 計画に含めよ。

## GATE-I3: フォールバック排除（Fail-Fast 強制）

### 基準 A（適合証明）
- すべてのエラー・異常状態に対し、即座に処理を
  中断しエラーを呼び出し元に伝播する実装であるか
- 入力値・前提条件の検証が処理の先頭で行われ、
  違反時に即座にエラーとなる実装であるか

### 基準 B（不適合排除）
- エラー発生時にデフォルト値で処理を続行する
  実装がないか
- 例外を catch して握りつぶす実装がないか
- 取得失敗時に空配列・空文字列・ゼロ値等で
  代替する実装がないか
- 条件分岐で「それ以外」を安全側に倒す
  実装がないか
- 本来の処理方式（LLM 呼び出し、外部 API 連携等）を
  省略し、簡易な代替処理で置き換える実装がないか
- プロンプト構築・API クライアント初期化等を
  行いながら実際には呼び出さず
  代替ロジックを使用する実装がないか

### FAIL 時の固定対応文

不当なフォールバック設計を検出。
フォールバックは絶対に許可しない。
設計から根本的に見直せ。

全てのエラー・異常状態で即座に処理を中断し
エラーを呼び出し元に伝播する設計にせよ。
デフォルト値での続行・例外の握り潰し・
代替値での処理継続を全て排除せよ。
