# RFC ゲートキーパー検証項目

## GATE-0: 仕様書改ざん排除

### 基準 A（適合証明）
- RFC ブランチ上でサービス仕様書（spec.md）が
  デフォルトブランチから変更されていないか

### 基準 B（不適合排除）
- サービス仕様書に対するいかなる差分も存在しないか

### FAIL 時の固定対応文

仕様書の改ざんを検出。
サービス仕様書がデフォルトブランチから変更されている。
AI による自動修正は禁止。
人間が差分を確認し、対応を判断せよ。

## GATE-1: E2E テスト品質

### 基準 A（適合証明）
- 各テストケースに外部サービスへの実接続が明記されているか
- テストデータがドメイン固有の実在データであるか
- テストシナリオがユーザ操作起点→最終アウトカム確認の
  構造であるか
- 各機能要件に対し正常系・異常系/境界値の2ケース以上
  あるか

### 基準 B（不適合排除）
- テストデータに汎用ダミーデータの兆候がないか
- 中間処理の確認で終わるテストケースがないか
- セットアップ手順に環境準備が欠落していないか

### FAIL 時の固定対応文

不当なテスト設計を検出。
モック・スタブ・ダミーデータを使った
E2E テストは絶対に許可しない。
テスト設計を根本的に見直せ。

全テストケースが以下を満たす状態にせよ:
- 外部サービス実接続
- ドメイン固有の実在データ使用
- ユーザ操作起点→最終アウトカム確認の構造
- 各機能要件に正常系・異常系/境界値の2ケース以上

## GATE-2: 勝手な先送り排除

### 基準 A（適合証明）
- 機能要件の全項目が設計・Task 計画で
  対応されている証拠があるか
- Task 計画が単一 PR で完遂可能な構成であるか

### 基準 B（不適合排除）
- 要件の無断除外・先送りの兆候がないか
- Task 計画の分割・段階化の兆候がないか

### FAIL 時の固定対応文

不当な先送り設計を検出。
勝手な先送りは絶対に許可しない。
設計から根本的に見直せ。

全機能要件を当 RFC のスコープとして
設計・Task 計画に含めよ。

## GATE-3: 環境状態変更タスクの欠落

### 基準 A（適合証明）
- 設計上必要な環境状態変更（DDL、cron、デプロイ設定、
  システムサービス設定、環境変数設定等）が
  Task 計画に含まれているか
- 環境状態変更が不要な場合、その根拠が明確か

### 基準 B（不適合排除）
- 設計に環境状態変更を要する要素があるにもかかわらず
  Task 計画に対応タスクが欠落していないか

### FAIL 時の固定対応文

設計上必要な環境状態変更
（DDL、cron、デプロイ設定、
システムサービス設定、環境変数設定等）を
全て Task 計画に含めよ。

## GATE-4: フォールバック排除（Fail-Fast 強制）

### 基準 A（適合証明）
- すべてのエラー・異常状態に対し、即座に処理を中断し
  エラーを呼び出し元に伝播する設計であるか
- 入力値・前提条件の検証が処理の先頭で行われ、
  違反時に即座にエラーとなる設計であるか

### 基準 B（不適合排除）
- エラー発生時にデフォルト値で処理を続行する設計がないか
- 例外を catch して握りつぶす設計がないか
- 取得失敗時に空配列・空文字列・ゼロ値等で代替する
  設計がないか
- 条件分岐で「それ以外」を安全側に倒す設計がないか
- リトライ・サーキットブレーカー等の耐障害パターンを
  ビジネスロジック層に組み込む設計がないか
- 本来の処理方式（LLM 呼び出し、外部 API 連携等）を
  省略し、簡易な代替処理で置き換える設計がないか
- プロンプト構築・API クライアント初期化等を
  行いながら実際には呼び出さず
  代替ロジックを使用する設計がないか

### FAIL 時の固定対応文

不当なフォールバック設計を検出。
フォールバックは絶対に許可しない。
設計から根本的に見直せ。

全てのエラー・異常状態で即座に処理を中断し
エラーを呼び出し元に伝播する設計にせよ。
デフォルト値での続行・例外の握り潰し・
代替値での処理継続を全て排除せよ。

## GATE-5: 絶対守ることの違反排除

※ §0 が空欄の場合、本ゲートは自動 PASS とする。

### 基準 A（適合証明）
- §0 の各項目に対して、RFC の設計が適合している
  積極的証拠があるか

### 基準 B（不適合排除）
- §0 の各項目に違反する設計がないか

### FAIL 時の固定対応文

§0 に定義された全項目に
適合する設計にせよ。

## GATE-6: E2E テスト仕様とサービス仕様書の整合性

### 適用条件

以下の両方を満たす場合のみ実行する。
いずれかを満たさない場合は自動 PASS とする。

1. サービス仕様書（docs/specs/*/spec.md）が存在する
2. サービス仕様書 §4.3 の RFC 一覧が1件のみである

### 基準 A（適合証明）
- サービス仕様書 §2 の全 E2E テスト要件 ID が
  RFC §7 のテストケースの「対応要件」列で参照されているか
- RFC §7 の各テストケースの「期待するアウトカム」が
  サービス仕様書 §2 の対応する「期待結果」と
  意味的に等価か

### 基準 B（不適合排除）
- サービス仕様書 §2 に存在する E2E テスト要件 ID のうち、
  RFC §7 でカバーされていないものがないか
- RFC §7 のテストケースがサービス仕様書 §2 の
  期待結果を緩和・簡略化していないか

### FAIL 時の固定対応文

E2E テスト仕様とサービス仕様書の不整合を検出。
RFC §7 のテストケースはサービス仕様書 §2 の
E2E テスト要件と完全に整合しなければならない。
サービス仕様書 §2 を Single Source of Truth として
RFC §7 を修正せよ。
